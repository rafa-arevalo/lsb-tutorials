Skills para o Micro-SaaS · 60 minutos

Sobre esta aula
AulaA06 — O Claude que trabalha no seu estilo
Duração60 minutos
ContextoMicro-SaaS para clínicas odontológicas — OdontoFlow
Entregável2 Skills criadas, testadas e ajustadas para o seu projeto

Você já percebeu que certas tarefas do projeto se repetem toda semana: analisar entrevistas e avaliar propostas de valor. Para cada uma você reescreveu um prompt longo — e vai continuar reescrevendo até o final do semestre se não fizer nada a respeito.

As Skills eliminam esse retrabalho. Uma Skill é um conjunto de instruções salvas que ensina ao Claude como executar uma tarefa específica do seu jeito. Você escreve uma vez. O Claude reproduz o padrão toda vez, sem variações indesejadas, em qualquer conversa futura.

Mas Skill não é solução para tudo — e parte do que você vai aprender nesta aula é justamente quando não vale criar uma.

01. O que é uma Skill — e o que ela não é

Um template inteligente que você escreve uma vez e reutiliza sempre.

Uma Skill é um conjunto de instruções salvas que define como o Claude deve executar um tipo específico de tarefa. Funciona como um template inteligente: você escreve o "modo de fazer" uma vez e o Claude aplica esse padrão automaticamente sempre que a situação se encaixar.

Sem SkillCom Skill
Reescreve o mesmo prompt toda vezEscreve uma vez, reutiliza sempre
Resultados inconsistentes entre sessõesOutput padronizado toda vez
Prompt se perde no histórico da conversaSkill salva em Configurações
Adequado para tarefas pontuaisAdequado para tarefas recorrentes

O que uma Skill não é

✕
Não é um agente autônomo. A Skill não age sozinha. Ela só é ativada quando você inicia uma conversa e o conteúdo corresponde ao gatilho.
✕
Não é memória. Ela define como o Claude executa, não o que ele sabe. Cada conversa ainda começa do zero.
✕
Não é substituta para bons prompts. Uma Skill com instruções ruins vai produzir resultados ruins de forma consistente — padroniza o problema junto com a solução.

02. Quando criar — e quando não criar

Três perguntas para decidir antes de configurar qualquer coisa.

Antes de criar uma Skill, responda as três perguntas abaixo. Skill faz sentido quando a resposta for "sim" para as três. Se falhar em qualquer uma, resolva manualmente por mais algumas rodadas antes de criar.

1
Essa tarefa se repete com o mesmo formato mais de 2 vezes?
Se é algo que você vai fazer uma vez só ou que muda muito de formato a cada vez, não vale criar.
2
O resultado precisa ser consistente entre sessões diferentes?
Se você quer que a décima análise de entrevista tenha exatamente a mesma estrutura da primeira, vale criar. Se cada análise pode ser diferente, não precisa.
3
Você consegue descrever o output ideal com precisão agora?
Se ainda não sabe exatamente como quer receber o resultado, criar uma Skill vai apenas padronizar algo que ainda não está bem definido. Faça a tarefa manualmente algumas vezes até entender o que realmente quer — depois crie a Skill.
❌ NÃO merece SkillTarefas inadequadas
  • Brainstorming de nomes para o produto (criativo, não estruturado)
  • Responder dúvidas pontuais sobre o projeto (cada dúvida é diferente)
  • Redigir um e-mail para uma situação única
  • Explorar uma ideia nova que você ainda está descobrindo
✅ MERECE SkillNo contexto do Micro-SaaS
  • Analisar toda entrevista de validação com a mesma estrutura
  • Avaliar cada versão da proposta de valor com os mesmos critérios
  • Revisar perguntas de entrevista antes de ir a campo

03. Anatomia de uma boa Skill

Três partes obrigatórias — e o gatilho é a mais crítica.

Uma Skill bem escrita tem três partes. O gatilho é a parte que mais falha quando mal escrita, por isso merece atenção especial.

Nome

Claro e descritivo. Você vai precisar identificá-la numa lista de várias Skills.

RuimVago demais
  • Análise
✅ BomDescritivo e específico
  • Análise de entrevista de cliente — OdontoFlow

Gatilho (descrição)

Define quando o Claude deve ativar a Skill. Quando você envia uma mensagem, o Claude lê a descrição de todas as suas Skills e decide se o conteúdo corresponde a alguma delas. Ele não busca palavras-chave exatas — ele interpreta se a situação descrita no gatilho corresponde ao que você enviou. Por isso:

✕
Gatilhos vagos ativam na hora errada, ou não ativam quando deveriam
✕
Gatilhos muito específicos deixam de ativar quando o input varia um pouco
✓
O melhor gatilho descreve a situação, não a palavra que você vai usar
TipoExemploProblema
Vago "Use quando eu pedir uma análise." "Análise" pode ser de código, de mercado, de texto. O Claude não sabe qual Skill ativar.
Específico demais "Use quando eu escrever 'analisar entrevista'." Se você colar uma transcrição sem essa frase exata, a Skill não ativa.
✅ Bem calibrado "Use quando eu colar notas brutas ou transcrição de uma entrevista com dono de clínica odontológica. O sinal: haverá falas diretas sobre dificuldades operacionais, geralmente entre aspas." Descreve a situação e o sinal no conteúdo — o Claude reconhece mesmo sem palavras exatas.

Instruções de output

Descrevem exatamente o que o Claude deve produzir: seções, formato, tom, nível de detalhe. Quanto mais específicas, mais consistente o resultado.

RuimVago
  • Analise o conteúdo e me dê um resumo útil.
✅ BomEspecífico
  • Gere um Artifact com 4 seções numeradas
  • Seção 1: dores com frequência e impacto
  • Seção 2: comportamento atual
  • Finalize com linha de conclusão obrigatória

04. Como criar uma Skill

Dois caminhos — escolha pelo seu ponto de partida.

Caminho 1 — Pelo painel

Use quando a tarefa for nova e você ainda está descobrindo o formato ideal. Escrever do zero te força a pensar com clareza sobre o que quer.

1
Acesse Configurações → Skills → Nova Skill
2
Preencha os três campos: Nome, Gatilho e Instruções de output
3
Salve. A Skill fica disponível em todas as conversas e Projects.

Caminho 2 — Pedindo ao Claude para criar

Use quando já tem um prompt testado que funciona bem. O Claude vai estruturar melhor do que você faria do zero.

Prompt — Caminho 2 Tenho usado este prompt com frequência e quero transformá-lo em Skill: [cole o prompt] Crie uma Skill com: - Nome claro e descritivo - Gatilho que descreve a situação e os sinais no conteúdo que indicam quando ativar (não use palavras-chave — descreva a situação) - Instruções detalhadas de output com seções numeradas e formato definido Formate pronto para colar no painel de Skills do Claude.
Qual caminho usar? Caminho 1 quando você ainda está descobrindo o formato ideal. Caminho 2 quando já tem um prompt testado — o Claude vai estruturar melhor do que você faria na mão. Na Demo 3 você vai ver o Caminho 2 sendo usado ao vivo.

05. Skill 1 — Análise de entrevista de cliente

Campos prontos para criar no painel · Configurações → Skills → Nova Skill

Usada toda vez que você colar notas ou transcrição de uma entrevista. Garante que cada análise siga a mesma estrutura, facilitando a comparação entre entrevistas ao longo do semestre.

🗣️ Nota do instrutor: Crie esta Skill antes de avançar para as Demos. Na Demo 1 e na Demo 2 ela será ativada ao vivo — você precisa tê-la salva para acompanhar o comportamento esperado.

Nome

Análise de entrevista de cliente — OdontoFlow

Gatilho

Use quando eu colar notas brutas ou transcrição de uma entrevista com dono de clínica odontológica para o projeto OdontoFlow. O sinal no conteúdo: haverá falas diretas do entrevistado sobre dificuldades operacionais ou de gestão de pacientes — geralmente entre aspas ou com marcação de nome e data.

Instruções

Ao receber notas ou transcrição de entrevista, gere um Artifact estruturado com as seguintes seções: 1. DORES IDENTIFICADAS Para cada dor mencionada, indique: - Nome da dor (em negrito) - Frequência: mencionada 1 vez / repetidamente / com ênfase emocional - Impacto percebido pelo entrevistado: alto / médio / baixo - Evidência: o que no texto indica esse impacto 2. COMPORTAMENTO ATUAL - Como o entrevistado resolve ou tenta resolver o problema hoje - Ferramentas, processos ou gambiarras mencionadas - Por que a solução atual falha ou é insuficiente 3. CITAÇÕES DIRETAS - Mínimo 3, máximo 5 trechos exatos das falas do entrevistado - Selecione os que melhor evidenciam as dores principais 4. HIPÓTESES PARA O PROJETO - O que essa entrevista confirma sobre o problema central - O que contradiz ou nuança a hipótese atual - Uma pergunta que levanta e vale investigar nas próximas Finalize com a linha de conclusão obrigatória: "Essa entrevista [confirma / contradiz / nuança] a hipótese principal porque [razão em 1 frase]."

06. Skill 2 — Avaliação de proposta de valor

Campos prontos para criar no painel · Configurações → Skills → Nova Skill

Usada toda vez que você quiser avaliar ou reformular a proposta de valor do OdontoFlow. Mantém os mesmos critérios de avaliação em todas as versões ao longo do semestre.

Nome

Avaliação de proposta de valor — OdontoFlow

Gatilho

Use quando eu descrever ou apresentar uma versão da proposta de valor do OdontoFlow e pedir avaliação, feedback ou comparação. O sinal no conteúdo: haverá um texto descrevendo o que o produto faz, para quem e qual benefício entrega — pode ser uma frase, um parágrafo ou um texto de apresentação.

Instruções

Ao receber uma proposta de valor, avalie e gere um Artifact com: 1. NOTAS POR CRITÉRIO Para cada critério abaixo, dê uma nota de 1 a 5 e uma justificativa de no máximo 1 linha: - Clareza para o usuário (o dono da clínica entende imediatamente?) - Diferenciação (é diferente do OdontoSoft, iClinic e genéricos?) - Alinhamento com as dores validadas (resolve o que os entrevistados disseram?) - Viabilidade (um estudante consegue construir em 3 meses?) 2. PONTOS FORTES O que está funcionando bem — máximo 3 pontos 3. GAPS CRÍTICOS O que está faltando ou confuso — máximo 3 pontos 4. PROPOSTA REFORMULADA Uma versão reescrita incorporando as melhorias — máximo 3 frases. Clara o suficiente para ser lida em 10 segundos por um dentista.
Com as duas Skills criadas, você está pronto para acompanhar as demonstrações e realizar a atividade prática.

07. Dados das Demonstrações

Transcrições e propostas usadas ao vivo nas 3 demos.

As transcrições são entrevistas com potenciais usuários do OdontoFlow, realizadas durante a fase de validação do problema. As propostas de valor são duas versões distintas — escritas com linguagens diferentes — para testar se a Skill 2 generaliza independentemente do estilo do texto.

Transcrição A
Dr. Rafael Mendes
Clínica com 2 dentistas · 11 anos · São Paulo — SP · 28 min
--- TRANSCRIÇÃO — DR. RAFAEL MENDES --- Data: [data da entrevista] Entrevistador: [nome do aluno] "Olha, o que mais me incomoda é o seguinte: eu faço um tratamento de canal, o paciente sai daqui aliviado, e a gente nunca mais vê ele. Aí seis meses depois eu descubro que ele foi fazer a coroa em outro lugar porque aquele consultório ficou na memória dele — não o meu. Isso dói, sabe? Não é só financeiro, é questão de confiança." "Minha recepcionista tem boa vontade, mas ela tem 40 outras coisas pra fazer. Confirmar consulta, atender telefone, organizar prontuário. O follow-up fica sempre pra depois. E 'depois' nunca chega." "Já tentei criar um grupo no WhatsApp com lembretes mensais. Mas aí você começa a misturar paciente com paciente, um responde no grupo do outro, vira bagunça. Tive que apagar. Agora mando mensagem individual mesmo, quando lembro." "O que eu precisaria é que o sistema olhasse pra mim e dissesse: Dr. Rafael, a Maria da Silva não vem desde março. Manda uma mensagem pra ela. Com o texto já pronto, personalizado. Eu só aprovaria e mandaria. Isso seria suficiente pra mudar o jogo aqui." "Preço? Me preocupo menos com preço do que com complexidade. Se eu precisar de um dia de treinamento pra usar, não vou usar. Tem que ser simples como o WhatsApp."
Transcrição B
Dra. Beatriz Almeida
Clínica solo · 4 anos · Campinas — SP · 35 min
--- TRANSCRIÇÃO — DRA. BEATRIZ ALMEIDA --- Data: [data da entrevista] Entrevistador: [nome do aluno] "Eu abri a clínica há 4 anos e nunca montei um processo de pós-atendimento. Simplesmente não sabia que precisava. Achava que se o paciente ficasse satisfeito, ele voltava naturalmente. Não volta. Você precisa aparecer." "Hoje eu sei que tenho uns 600 pacientes cadastrados no sistema. Mas ativos mesmo — que vieram nos últimos seis meses — devem ser uns 180. O restante eu nem sei se ainda mora na cidade." "O que faço hoje: quando tenho buraco na agenda, peço pra minha auxiliar ligar pra uma lista de pacientes que não vêm há muito tempo. Ela passa uma tarde ligando, uns 30% atendem, metade desses agenda. Funciona, mas é um processo sofrido. Ninguém gosta de ligar pra lista fria." "Já tentei um CRM. Era da área de vendas, não de saúde. Os campos não faziam sentido pra clínica — tinha 'lead', 'oportunidade', 'pipeline'. Eu queria saber quando a paciente fez a última limpeza, não em que estágio do funil ela estava. Desisti em duas semanas." "O ideal seria algo que já viesse com o fluxo montado para clínica odontológica. Não quero configurar nada do zero. Quero abrir, colocar o nome dos pacientes e começar a usar no mesmo dia."
Proposta A
Linguagem técnica e orientada a produto
OdontoFlow é uma plataforma de gestão de relacionamento com pacientes desenvolvida especificamente para clínicas odontológicas de pequeno porte. O sistema monitora automaticamente o intervalo entre consultas de cada paciente e aciona fluxos de reengajamento via WhatsApp quando o período sem retorno ultrapassa o limiar configurado pela clínica — padrão de 90 dias. As mensagens são geradas com personalização por nome, tipo de tratamento realizado e tempo de ausência. Integração nativa com as principais agendas digitais do mercado. Implantação em até 60 minutos sem necessidade de suporte técnico.
Proposta B
Linguagem direta e orientada ao problema do usuário
Sua clínica tem pacientes que deveriam ter voltado e não voltaram. O OdontoFlow identifica quem são essas pessoas e escreve a mensagem certa para cada uma — você só lê, ajusta se quiser, e manda no WhatsApp. Sem planilha. Sem ligar pra lista fria. Sem treinar recepcionista. Funciona no primeiro dia. Se você souber usar o WhatsApp, sabe usar o OdontoFlow.

08. Demonstrações + Prática

3 demonstrações ao vivo, atividade prática com 4 passos e entregável.

Demo 1 Skill 1 em ação: análise sem escrever nenhuma instrução ~8 min

Abra o Claude.ai, acesse o Project do OdontoFlow e inicie uma nova conversa. Cole apenas a Transcrição A do Dr. Rafael — sem escrever nenhuma instrução antes ou depois do texto.

O que observar

O Claude reconhece o gatilho da Skill 1 e entrega o Artifact com as 4 seções estruturadas automaticamente. Você não pediu nada — só colou o conteúdo.

🗣️ Nota do instrutor: Aponte para a ausência de qualquer instrução na mensagem enviada. Só o conteúdo da transcrição. Isso é o comportamento esperado de uma Skill bem escrita: ela ativa pela situação, não pela palavra.
Demo 2 Com Skill vs. sem Skill — e o teste de generalização ~12 min

Esta é a demonstração mais importante da aula — observe com atenção.

Parte A — Comparação direta

Abra uma conversa comum fora do Project e cole a mesma Transcrição A do Dr. Rafael, também sem nenhuma instrução. Compare as duas respostas lado a lado:

✕
Fora do Project: texto corrido, sem as 4 seções, provavelmente sem Artifact
✓
Dentro do Project com Skill: estrutura definida, Artifact utilizável, mesmas seções toda vez

A diferença não é estética. Sem a Skill, cada análise feita ao longo do semestre vai ter uma estrutura diferente. Com a Skill, a décima análise vai ter exatamente a mesma estrutura da primeira. Isso importa quando você tem 8 entrevistas para comparar numa matriz de priorização.

Parte B — Teste de generalização

Ainda dentro do Project, abra uma nova conversa e cole apenas a Transcrição B da Dra. Beatriz — sem nenhuma instrução.

A Skill foi construída para entrevistas com donos de clínicas, mas o Dr. Rafael e a Dra. Beatriz têm perfis diferentes: ele tem 11 anos de mercado e clínica com 2 dentistas; ela é solo com 4 anos. As falas têm tom e vocabulário distintos. Se o gatilho foi bem escrito, a Skill deve ativar e produzir o mesmo Artifact estruturado para as duas entrevistas.

🗣️ Nota do instrutor: Este é o critério real de qualidade de uma Skill: ela funciona para qualquer instância da mesma tarefa, não só para o exemplo que você usou ao criá-la. Se a Skill não ativou para a Dra. Beatriz, o gatilho estava específico demais — descreve palavras de um entrevistado específico, não a situação geral.
Demo 3 Criando a Skill 2 pelo Caminho 2 — e testando a generalização ~15 min

Em vez de criar a Skill 2 do zero, veja como o Claude pode criá-la a partir de um prompt já existente. Abra uma conversa dentro do Project e envie:

Tenho usado este prompt repetidamente para avaliar versões da proposta de valor do OdontoFlow: "Avalie esta proposta de valor considerando: (1) clareza para o usuário de 1 a 5, (2) diferenciação em relação a OdontoSoft e iClinic de 1 a 5, (3) alinhamento com a dor de follow-up manual de 1 a 5, (4) viabilidade para um estudante construir em 3 meses de 1 a 5. Para cada critério: nota e justificativa de 1 linha. Ao final: 2 pontos fortes, 2 gaps críticos e a proposta reescrita incorporando as melhorias." Transforme em uma Skill estruturada com: - Nome claro e descritivo - Gatilho que descreve a situação e os sinais no conteúdo que indicam quando ativar (não use palavras-chave — descreva a situação) - Instruções detalhadas de output com seções numeradas Formate pronto para colar no painel de Skills do Claude.
Antes de salvar — leia o gatilho gerado

Ele descreve uma situação concreta ou apenas palavras que você poderia digitar? Se for só palavras, reescreva o gatilho antes de salvar.

Após salvar — faça os dois testes em sequência

Teste 1: Cole a Proposta A (linguagem técnica — "plataforma", "fluxos de reengajamento", "limiar configurado")

Teste 2: Cole a Proposta B (linguagem direta — "sua clínica tem pacientes que deveriam ter voltado")

🗣️ Nota do instrutor: Se a Skill ativou para as duas propostas, o gatilho está calibrado. Se ativou apenas para a A, ele estava preso à linguagem técnica do primeiro exemplo — precisa ser reescrito para descrever a situação em vez das palavras.

Atividade Prática

Regra desta atividade: você vai escrever os gatilhos do zero, sem copiar os exemplos da demonstração. Essa é a habilidade que mais diferencia uma Skill que funciona de uma que falha — e ela só se desenvolve praticando.
1
Avalie quais tarefas do seu projeto merecem uma Skill.
Aplique as três perguntas para cada tarefa recorrente do seu projeto:
(a) Essa tarefa vai se repetir com o mesmo formato mais de 2 vezes?
(b) O resultado precisa ser consistente entre sessões?
(c) Você consegue descrever o output ideal com precisão agora?
Anote as que passam nas três. Para as que falharam, continue fazendo manualmente por mais algumas rodadas.
2
Crie a Skill de análise de entrevista do seu projeto.
Acesse Configurações → Skills → Nova Skill. Para o gatilho, escreva respondendo: qual é a situação? e o que vai aparecer no conteúdo que sinaliza esse momento? Não use palavras-chave.
Depois, faça 2 testes: uma entrevista que você já realizou e outra com um entrevistado diferente. Se a estrutura saiu igual nos dois, a Skill está generalizando bem.
3
Crie a Skill de avaliação de proposta de valor.
Mesma lógica do Passo 2. Escreva o gatilho do zero descrevendo a situação. Defina os critérios de avaliação que fazem sentido para o seu segmento — não precisam ser os mesmos do OdontoFlow.
Teste com a proposta atual do projeto e com uma versão reformulada em linguagem diferente (mais formal, mais coloquial — tanto faz).
4
Ciclo de ajuste.
Escolha a Skill que produziu o resultado menos satisfatório e use o prompt abaixo para corrigi-la:
Prompt — Ciclo de ajuste Ciclo de ajuste Esta é minha Skill: Nome: [nome] Gatilho: [gatilho] Instruções: [instruções] Testei com este input: [cole o input que gerou o problema] O problema foi: [descreva com precisão o que falhou — a Skill não ativou? O output saiu sem alguma seção? O formato estava diferente do esperado?] Corrija apenas a parte problemática, mantendo o que funcionou no primeiro teste.

✓
