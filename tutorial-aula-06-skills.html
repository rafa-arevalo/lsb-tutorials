Uma Skill é um atalho de comportamento que você configura uma vez e aciona com uma frase. Sem reescrever prompt. Sem repetir instruções. O Claude reconhece o gatilho e executa — sempre da mesma forma, sempre com o mesmo padrão.

01. O que é uma Skill — e o que ela não é

Entenda o conceito antes de criar.

Uma Skill é um atalho de comportamento persistente. Você define uma vez: um nome, uma condição de disparo (o gatilho) e o que o Claude deve fazer. Quando você envia uma mensagem que corresponde ao gatilho, o Claude executa automaticamente — sem você precisar repetir as instruções.

Analogia

Pense em uma macro no Excel. Você grava o padrão uma vez. Depois, basta um atalho de teclado para executar. A Skill funciona igual: você configura o comportamento uma vez, e o gatilho é o atalho.

O que uma Skill NÃO é

✕
Não é uma conversa salva. Skills não guardam histórico de mensagens anteriores. Cada conversa ainda começa do zero — a Skill só define como o Claude se comporta dentro da conversa.
✕
Não é um Project. Skills não armazenam documentos nem criam memória permanente sobre o seu projeto. Para isso você usa o Project com instruções e arquivos (Aula 04).
✕
Não é um prompt que você cola toda vez. A diferença é que você não precisa escrever as instruções a cada conversa — o Claude já sabe o que fazer assim que reconhece o gatilho.

Skill vs. Instrução no Project

Instrução no ProjectComportamento geral
  • Sempre responder em português
  • Usar formato de lista com múltiplos itens
  • Adotar tom direto e objetivo
  • Aplicado em toda e qualquer conversa
SkillComportamento específico
  • Acionado por um gatilho definido
  • Executa estrutura específica de análise
  • Pode existir múltiplas Skills paralelas
  • Só dispara quando o gatilho aparece

02. Quando criar — e quando não criar

Três perguntas para decidir se vale o esforço de configurar.

Antes de criar uma Skill, passe pelas três perguntas abaixo. Se responder sim para pelo menos duas, vale a pena criar.

1
Você repete essa tarefa mais de uma vez por semana?
Se for uma tarefa rara ou única, um bom prompt direto na hora resolve melhor e com menos configuração.
2
A tarefa tem um padrão claro de entrada e saída?
Se a estrutura da resposta esperada varia muito a cada vez, a Skill fica rígida demais — você vai precisar reescrever de qualquer forma.
3
Você já tem um prompt que copia em conversas diferentes?
Esse é o sinal mais direto. Se você já tem um prompt que funciona e só copia ele, esse prompt deve virar Skill.

Quando NÃO criar uma Skill

Evite criar Skills quando:
· A tarefa é esporádica ou única — prefira um prompt direto
· O pedido é vago demais ("me ajude com meu projeto") — sem padrão de entrada definido, o gatilho não pode ser calibrado
· Você ainda está descobrindo o formato ideal da resposta — espere estabilizar antes de configurar

Exemplos do projeto

TarefaCriar Skill?Por quê
Analisar transcrição de entrevista ✅ Sim Repetitiva, padrão claro, mesmas seções toda vez
Avaliar proposta de valor ✅ Sim Critérios fixos, reutiliza em múltiplos rascunhos
Responder e-mail pontual do grupo ❌ Não Contexto único, não generaliza para outros e-mails
"Me ajude a pensar no problema" ❌ Não Sem padrão de entrada — gatilho impossível de calibrar

03. Anatomia da Skill

Os três campos e como calibrar o gatilho certo.

Toda Skill tem exatamente três campos. O mais crítico é o Gatilho — é ele que determina se a Skill vai disparar no momento certo ou nunca disparar.

Campo 1 — Nome
Identificação interna da Skill. Use um nome descritivo que você vai reconhecer na lista. Ex: "Análise de Entrevista com Cliente"
Campo 2 — Gatilho
A condição que aciona a Skill. O Claude monitora suas mensagens e, ao identificar o gatilho, executa as instruções automaticamente. Ex: "Quando eu enviar uma transcrição de entrevista com potencial cliente"
Campo 3 — Instruções
O que o Claude deve fazer quando o gatilho é ativado: o comportamento esperado, o formato de resposta e o nível de detalhe. Ex: "Identifique as principais dores, jobs-to-be-done, objeções, disposição para contratar e gere 3 insights..."

Calibrando o gatilho

O gatilho precisa ser específico o suficiente para não disparar por acidente, mas genérico o suficiente para generalizar para casos futuros. Veja a diferença:

TipoExemploPor que não funciona
Vago "análise" Claude não sabe quando acionar — pode disparar em qualquer pedido de análise, mesmo sem ser entrevista
Específico demais "Analise a entrevista do Dr. Marcos Silva feita em 12/04" Nunca vai disparar para outros entrevistados ou outras datas — inutilizável
✅ Bem calibrado "Quando eu enviar uma transcrição de entrevista com potencial cliente" Generaliza para qualquer entrevista futura do projeto — exatamente o que queremos
Regra prática: escreva o gatilho pensando em "para quais casos futuros eu quero que isso dispare?" Se a resposta cobrir os próximos 10 usos com natureza similar, o gatilho está bem calibrado.

04. Como criar uma Skill

Dois caminhos: manual ou gerando via Claude.

Caminho 1 — Criar manualmente

1
Abra o Claude.ai e clique no seu nome ou avatar no canto inferior esquerdo da tela.
2
Acesse Settings → Skills (ou "Habilidades", dependendo do idioma da interface).
3
Clique em "Create new Skill" e preencha os três campos: Nome, Gatilho e Instruções.
4
Clique em Save. A Skill fica disponível em todas as conversas — inclusive dentro do seu Project.

Caminho 2 — Pedir para o Claude gerar a Skill

Se você já tem um prompt que funciona, pode pedir para o Claude transformá-lo em uma Skill completa. Cole o prompt abaixo em uma conversa nova, substituindo [COLE SEU PROMPT AQUI]:

Prompt — Caminho 2 Quero criar uma Skill no Claude a partir do seguinte prompt que já uso: [COLE SEU PROMPT AQUI] Por favor, gere para mim: 1. Um Nome para a Skill (curto e descritivo) 2. Um Gatilho bem calibrado — específico o suficiente para não disparar por acidente, genérico o suficiente para funcionar em casos futuros similares 3. Instruções completas — o que o Claude deve fazer, em qual formato e com qual nível de detalhe Formate a resposta nos três campos separados, prontos para eu copiar direto no formulário de criação da Skill.
Qual caminho usar? Se você já tem um prompt testado e funcionando, use o Caminho 2 — o Claude vai gerar um gatilho mais preciso do que você escreveria na mão. Se está criando do zero, use o Caminho 1 com os campos das seções 05 e 06 (já prontos para copiar).

05. Skill 1 — Análise de Entrevista

Campos prontos para copiar e configurar no Claude.

Esta Skill será acionada toda vez que você enviar uma transcrição de entrevista. Ela devolve uma análise estruturada no mesmo formato — sem você precisar reescrever o prompt a cada nova entrevista.

🗣️ Nota do instrutor: Você vai usar esta Skill nas Demos 1 e 2. Antes de criá-la, leia os campos abaixo e entenda cada parte — especialmente o gatilho e o que as instruções pedem nas 7 seções.

Campo 1 — Nome

Análise de Entrevista com Cliente

Campo 2 — Gatilho

Quando eu enviar uma transcrição de entrevista com potencial cliente

Campo 3 — Instruções

Analise a transcrição de entrevista com potencial cliente e estruture a resposta nas seguintes seções: 1. PERFIL DO ENTREVISTADO Nome, cargo/área e contexto relevante identificado na conversa. 2. DORES PRINCIPAIS Liste as 3 a 5 principais dores mencionadas (explícitas ou implícitas). Para cada dor, indique se foi mencionada de forma direta ou inferida do contexto. 3. JOBS-TO-BE-DONE O que o entrevistado está tentando realizar? Liste os trabalhos funcionais, emocionais e sociais identificados. 4. OBJEÇÕES E BARREIRAS O que poderia impedir essa pessoa de adotar uma solução? Liste objeções explícitas e barreiras inferidas. 5. DISPOSIÇÃO PARA CONTRATAR Escore de 1 a 5 (1 = muito baixa, 5 = muito alta) com justificativa baseada em evidências da transcrição. 6. SÍNTESE EM UMA FRASE Uma frase que captura o essencial desta entrevista para quem não a leu. 7. TOP 3 INSIGHTS Os três insights mais importantes para o projeto, em ordem de relevância.
Pronto para criar? Abra Claude.ai → Settings → Skills → Create new Skill. Cole cada campo acima. Na seção 07 (Dados Demo) você encontra as transcrições para testar após criar.

06. Skill 2 — Avaliação de Proposta de Valor

Campos prontos para copiar e configurar no Claude.

Esta Skill avalia qualquer proposta de valor que você escrever — com os mesmos critérios toda vez. Útil ao longo de todo o semestre, à medida que você itera e refina a proposta do projeto.

Campo 1 — Nome

Avaliação de Proposta de Valor

Campo 2 — Gatilho

Quando eu pedir para avaliar uma proposta de valor

Campo 3 — Instruções

Avalie a proposta de valor fornecida usando os seguintes critérios: 1. CLAREZA (1 a 10) A proposta é compreensível para alguém que não conhece o projeto? O problema resolvido fica claro em menos de 10 segundos de leitura? 2. ALINHAMENTO COM DORES (1 a 10) A proposta se conecta com dores reais identificadas em entrevistas? Cite a dor específica que ela endereça. 3. DIFERENCIAÇÃO (1 a 10) O que torna essa proposta diferente de soluções já existentes? Há um elemento de singularidade identificável? 4. LINGUAGEM DO CLIENTE (1 a 10) A proposta usa as palavras que o cliente usaria para descrever seu próprio problema? Ou usa jargão interno do projeto? 5. PONTUAÇÃO GERAL Média dos quatro critérios com uma frase de justificativa. 6. TOP 3 MELHORIAS SUGERIDAS As três mudanças mais impactantes, da mais urgente à menos urgente. Para cada uma: o problema, a sugestão de mudança e um exemplo da frase reescrita.

07. Dados das Demonstrações

Transcrições e propostas para usar nas demos ao vivo.

🗣️ Nota do instrutor: Estes são os dados que você vai usar nas 3 demonstrações da próxima seção. Leia rapidamente para entender o perfil de cada entrevistado antes das demos.
Transcrição 1 Dr. Marcos Oliveira — Gestor de Projetos
entrevista-marcos-oliveira.txt
Entrevistador: Obrigado por aceitar conversar com a gente, Dr. Marcos. Para começar, como você descreveria o maior desafio no seu dia a dia como gestor de projetos? Dr. Marcos: Olha, o principal problema é o desalinhamento entre times. A gente começa um projeto com uma expectativa, e duas semanas depois cada área está interpretando o escopo de um jeito diferente. O cliente pede uma coisa, o dev entende outra, e o comercial vendeu uma terceira. Quando percebemos o desvio, já perdemos tempo e a confiança do cliente vai junto. Entrevistador: Com que frequência isso acontece? Dr. Marcos: Mais do que eu gostaria de admitir. Diria que em uns 60% dos projetos temos pelo menos um episódio sério de desalinhamento. E o custo é alto — retrabalho, reuniões extras, às vezes atraso. Entrevistador: O que você já tentou para resolver isso? Dr. Marcos: Já tentamos de tudo: Notion, Asana, reuniões semanais. O problema não é falta de ferramenta. É que ninguém tem tempo de atualizar o status em tempo real. A documentação vira um fantasma que todo mundo sabe que existe mas ninguém consulta. Entrevistador: Se você pudesse ter uma solução ideal, como seria? Dr. Marcos: Algo que registrasse automaticamente o que foi decidido em reunião e mandasse um resumo para todos os envolvidos — sem eu precisar fazer ata. Se isso existisse e fosse confiável, eu pagaria por isso sem pensar. Entrevistador: Você já pesquisou soluções nesse sentido? Dr. Marcos: Já vi algumas ferramentas de transcrição, mas elas transcrevem tudo e jogam o texto bruto. Aí preciso de meia hora para extrair o que importa. Quero o resumo executivo, as decisões e os próximos passos — não a transcrição completa.
Transcrição 2 Dra. Carla Mendes — Head de Produto
entrevista-carla-mendes.txt
Entrevistador: Dra. Carla, qual é o maior ponto de atrito no seu trabalho hoje? Dra. Carla: Priorização. Tenho dezenas de demandas chegando ao mesmo tempo — do comercial, do suporte, dos usuários, da liderança. O problema não é não ter dados; é que cada área apresenta seus dados de um jeito diferente e fico tentando comparar coisas incomparáveis. Entrevistador: Como você decide o que entra no sprint? Dra. Carla: Honestamente? Na base da intuição e de quem gritou mais alto na última semana. Não é ideal. Já tentei criar frameworks de priorização com o time, mas ninguém preenche de forma consistente. Cada PM tem seu jeito e isso impossibilita a comparação. Entrevistador: O que mais te frustra nesse processo? Dra. Carla: O tempo que gasto explicando para a liderança por que priorizamos X e não Y. Toda reunião de roadmap vira uma negociação política, não uma conversa baseada em evidência. Se eu tivesse um documento que consolidasse tudo automaticamente, com critérios transparentes, seria muito mais fácil defender as decisões. Entrevistador: Você pagaria por uma ferramenta que resolvesse isso? Dra. Carla: Dependeria de quanto. Se economizasse umas 4 horas por semana — que é o que esse processo consome de mim hoje — certamente valeria. Mas teria que ser simples — qualquer coisa que exija mais de 10 minutos de configuração por semana morre na adoção.
Proposta Formal Versão 1 — Linguagem técnica e corporativa
proposta-formal.txt
Nossa solução oferece uma plataforma integrada de gestão e rastreabilidade de decisões para equipes de produto e projetos, eliminando o desalinhamento comunicacional e otimizando o fluxo de informações entre stakeholders. Por meio de inteligência artificial aplicada ao processamento de linguagem natural, o sistema captura automaticamente os artefatos decisórios gerados em reuniões e os distribui de forma estruturada para os envolvidos, garantindo governança, rastreabilidade e redução de retrabalho.
Proposta Coloquial Versão 2 — Linguagem simples e direta
proposta-coloquial.txt
Chega de sair de reunião sem saber o que foi decidido. Nossa ferramenta escuta a reunião, identifica as decisões e os próximos passos e manda um resumo direto para quem precisa saber — sem ninguém precisar fazer ata. Em vez de ficar tentando lembrar o que foi combinado, você tem tudo registrado e acessível em segundos.

Passo 1 — Abra uma conversa nova (fora do Project)

Passo 2 — Cole a transcrição do Dr. Marcos

Demo 2 Com vs. sem Skill — teste de generalização ~10 min

Objetivo: mostrar a diferença entre ter e não ter a Skill, e confirmar que ela generaliza para uma segunda entrevista (Dra. Carla) sem ajuste.

Parte A — Sem Skill

Abra uma conversa em uma conta que não tenha a Skill configurada (ou peça para um aluno abrir no próprio computador). Cole a transcrição da Dra. Carla sem nenhum prompt. Mostre o resultado: o Claude responde de qualquer jeito, sem estrutura definida.

Parte B — Com Skill

Agora cole a mesma transcrição da Dra. Carla em uma conversa com a Skill ativa.

Segue a transcrição da entrevista com a Dra. Carla Mendes, head de produto. [cole aqui o conteúdo de entrevista-carla-mendes.txt da seção 07]
🗣️ Nota do instrutor: O ponto central desta demo é o teste de generalização. A Skill foi criada com a transcrição do Dr. Marcos em mente — mas ela disparou igualmente para uma entrevistada diferente, de área diferente, com problemas diferentes. É isso que torna uma Skill útil: ela generaliza.
Demo 3 Criar Skill 2 via Claude + teste de generalização ~12 min

Passo 3 — Teste com Proposta Formal
Por favor, avalie a seguinte proposta de valor: [cole aqui o conteúdo de proposta-formal.txt da seção 07]
Passo 4 — Teste com Proposta Coloquial

Atividade Prática

1
Ative a Skill 1 na sua conta (Settings → Skills). Use os campos da seção 05. Se já criou durante a demo, confirme que está ativa.
2
Ative a Skill 2 usando o Caminho que preferir — manual (seção 06) ou via Claude (prompt do Caminho 2, seção 04).
3
Teste a generalização: cole uma das transcrições do seu próprio projeto (se já tiver) ou use as da seção 07. Verifique se as 7 seções aparecem e se fazem sentido para o perfil do entrevistado.
4
Faça o ciclo de ajuste: se alguma seção da análise ficou vaga ou incompleta, use o prompt abaixo para refinar as instruções da Skill sem recriar do zero.
Prompt — Ciclo de ajuste Ciclo de ajuste Estou ajustando as instruções de uma Skill de análise de entrevistas. O problema que identifiquei foi: [descreva o que ficou incompleto ou vago na análise] As instruções atuais são: [cole as instruções atuais da Skill] Por favor, reescreva apenas a seção problemática das instruções para corrigir o ponto identificado, mantendo o restante intacto.
